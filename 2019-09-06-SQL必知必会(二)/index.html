SQL必知必会(二)

发表于 2019-09-06 | 更新于: 2019-09-06 |
字数统计: 1.1k | 阅读时长 ≈ 3

前言

每天都要学习一点新技术，更新自己的技术库

数据表中的那些键

  • 超键：能唯一标识元组的属性集叫做超键
  • 候选键：最小超键，即不包含其他属性。例如该属性集中找不到其真子集满足唯一标识元组的条件
  • 主键：候选键中选一个作为主键
  • 外键：某表A中的属性集不是该表的主键，是另外一个表B的主键
  • 主属性：在候选键中的属性称为主属性
  • 非主属性：与主属性相对的，不包含在任何一个候选键中的属性

通常，我们也将候选键称为“码”，主键也称为“主码”

举个例子：球员表 定义为 球员编号、姓名、身份证号、年龄和球队编号

超键即为 属性集包含 球员编号 或者身份证

候选码 （球员编号) （身份证)

主键 从候选码中任意选择一个

外键 球队编号

从1NF到3NF

1NF

指的是数据库表中任何属性都是原子性，不可再分。就是不能对字段进行拆分

2NF

指数据表中的非主属性都要和这个数据表中的候选键有完全依赖关系

举个例子

一张球员比赛表 player_game包含 球员编号、姓名、年龄、比赛编号、比赛时间和比赛场地

候选键和主键都是(球员编号、比赛编号)

所以(球员编号、比赛编号)->(姓名、年龄…)

但是不满足第二范式

(球员编号)->(姓名、年龄)

(比赛编号)->(比赛时间、比赛场地)

这样就说明 非主属性并非完全依赖于候选键会造成数据冗余等错误

  • 数据冗余 如果一个球员参加M场比赛，其姓名和年龄就重复了m-1次，一场比赛有n个球员参加，其比赛时间和地点就会重复n-1次
  • 插入异常 如果新加一场比赛，但是由于参加队员未知，所以就无法插入
  • 删除异常 删掉某个球员编号 就会将比赛信息删掉
  • 更新异常 如果更新某场比赛时间，那么就要调整所有数据

所以应该设计成三张表

球员 包含球员编号 年龄属性等

比赛 包含比赛编号 比赛时间等

球员比赛关系 球员编号 比赛编号 得分属性

理解一下：2NF就是然每一张表只做一件事

3NF

非主属性只能依赖于候选键

例如球员表中 球员编号 球队名称 球队教练

候选键为 球员编号

球员编号决定了球队名称，球队名称又决定了球队教练。所以就会出现传递依赖。

如何解决？

球员表 球员编号、球队名称

球队表 球队名称 球队教练

小结

1nf一般DBMS都已经做到了

2nf 只需要判断非主属性中是否有只要候选码的真子集就可以确定的

3nf 判断非主属性中是否可以决定的内容

不过将表拆分成这样，使得查询的效率实在是低

实战

学生选课表 属性 学号、姓名、课程名称、分数、系别和系主任

学生表：学号(候选码) 姓名 系别(外键)

系表：系别(候选码) 系主任

课程表：课程编号(候选码) 课程名称

学生-课程表：学号 +课程编号(候选码) 分数

BCNF巴斯范式

例如有张表 仓库名、管理员、物品名、数量

候选键为(仓库名、物品名)(管理员、物品名)

满足1nf,因为数量是完全依赖于候选键的满足2nf，因为不存在传递候选键依赖满足3nf

存在问题

  • 增加一个仓库，物品为空。根据数据库的实体完整性会出现插入异常(主键不能为空)
  • 仓库更换了管理员，更改多条记录
  • 如果物品为空，该仓库信息会被删除掉

根据BCNF要求拆分成

仓库表:（仓库名、管理员)

库存表：(仓库名、物品名、数量)

反范式设计

如果需要多个表查询，可以尝试用空间换时间，即数据冗余

例如实战中不断增加的pic_url，减少查询

后记

没有绝对的标准，只有最适合的。需要不断去摸索，不断总结。
