虚拟化与容器

虚拟这个词其实比较常见的，例如虚拟机啥的。所以我们应该并不陌生。

虚拟化是一种将计算机物理资源进行抽象，转为虚拟的资源供程序使用

简单说就是，将底层抽象。这个带来的好处就是：不用去操心底层的具体情况，程序不需要是适配不同的底层。

早在IBM公司为了解决底层的接口不同，提出了虚拟化的技术，从而实现程序跨平台这一功能。

虚拟化在资源管理方面能够提高资源的利用率，但是不会减少资源的占用率。利用率和占用率是不同的两个概念，所以要辨别好。

那么虚拟化是如何提高资源的利用率呢？

虽然我们能够在一个操作系统里面运行多个程序，但是前提是程序的本身不存在冲突，例如不能使用同一个端口等。虚拟化技术就是把程序隔离在不同环境里，既然环境不同，那么所运行的程序自然也就不会产生冲突。

接着来看虚拟机，虚拟机虽然屏蔽了底层。例如Java的JVM虽然能够使代码编写一次，到处运行【在开发过程中，无需关注不同系统提供的API】。但是虚拟机有个缺点就是所有指令都需要经过虚拟机监视器的处理才转为机器所能够识别的机器码。

容器技术是一种全新的虚拟化技术。

容器化是这操作系统自身提供的一些接口，能够让应用程序间能够互不干扰的运行

容器相对于虚拟机来说就是，容器根本不对指令进行转换，所以在性能上，容器要远远好过虚拟机。

Docker技术实现

Docker技术的实现有三大技术，包括命名空间(Namespaces)、控制组(Control Groups)、联合文件系统(Union File System)

命名空间Namespaces

命名空间我第一个想到的是C++中的using namespaces，在编程语言中Namespaces主要是为了划分不同模块，区别不同模块中相同的命名。

在操作系统也有相同的实现，能够将计算机资源进行切割和划分，形成各自独立的空间。例如PID Namespaces表示可以造就一个独立的进程运行空间，在该空间里，进程号又可以从1开始，不会感知到外界系统的其他进程或者其他命名空间中运行的程序。

Docker利用PID Namespaces实现了容器中隔离程序中运行的进程隔离。

控制组Control Groups

CGroups用于控制计算机资源，与Namespaces不同的时，控制的是硬件资源。同时还有分配资源的作用

分配资源是很重要的，可以指定某个隔离环境对任意资源的占用值和利用率。在实际生产环境中很重要，因为在实际生产环境中会有业务系统和监控系统，不希望监控系统占用的资源过多。

联合文件系统Union File System

联合文件系统是指能够同时挂载不同实际文件或文件夹到同一个目录下，从而实现联合文件的结构

联合文件系统不属于虚拟化技术，但是Docker却将其引入，从而实现Docker的快启快停的目的，并且不会占用太多的文件系统。

在Git中，我们虽然每个版本都commit，但是文件占用却没有增加，就是因为使用了联合文件系统。

docker理念

Docker推崇的是一种轻量级容器结构，例如要搭建一套LAMP结构。在docker中的最佳实践就是分别基于Apach、Mysql、Php三个镜像建立容器。然后分别运行。

所以自己一开始中的mysql啥的都是一个镜像一个容器做法符合Docker理念

四大组成对象

image镜像

镜像指的是虚拟环境中运行的原始的文件系统

docker基于联合文件系统，创建了一种增量式的镜像结构

每次对镜像的修改都会将以上修改成一个新的镜像层。所以这种结构，镜像实质上是无法修改的，每次修改都会产生新的镜像。

Container容器

在docker中，容器就是用来隔离虚拟环境的基础设施。

如果把镜像理解为类，那么容器就是实例

Network网络

网络在分布式系统中非常常见，用来在程序间作为信息或数据的交换。但是现在我对docker提供强大的网络还不是很了解

数据卷

在Docker中，将数据共享或持久化的文件和目录称为数据卷

docker镜像

查看镜像

通过执行docker images可以查看当前存放和管理的镜像

可以查看镜像ID，构建时间，占用空间

镜像命名

镜像命名

  • username:主要用于识别上传镜像的不同用户
  • repository:用于识别进行的内容
  • tag:表示镜像的版本，用来区分进行内容的不同细节

另外docker镜像缺省的时候会用latest标签，所以在我们不需要软件的迭代周期，就可以下载软件的最新版。

镜像仓库

类似于git一样，镜像仓库就是用来托管镜像。借助镜像仓库，我们得到一个镜像的中转站，可以把开发环境所使用的镜像推送到镜像仓库，然后在测试或者生产环境上拉取它们。

拉取镜像

docker pull拉取镜像，会先下载该镜像所需的所有镜像层，然后组装。如果本地有的话，就会跳过该镜像的下载。

搜索镜像

可以通过docker hub来搜索镜像。也可以通过执行docker search images

管理镜像

通过docker inspect查看对应镜像的信息

参数识别

只要我们传入的参数能够表示对应的镜像就可以，不需要传入多余的参数

删除镜像

删除镜像的命令是docker rmi

容器生命周期

  • created表示容器已经被创建，容器所需的相关资源已经准备好了，但是容器中的程序还没有运行
  • running表示容器正在运行，即容器中的程序正在运行
  • paused表示容器已暂停，容器中所有程序都处于暂停状态
  • stopped表示容器已经停止，所占用的资源和沙盘依然存在
  • deleted表示容器已经被删除

创建容器

选择好镜像后可以通过执行docker create来创建容器，但是如果每次都是通过容器ID来操作容器显得很繁琐，所以可以通过容器名来操作容器。

在创建容器的时候，可以通过--name来配置容器名

非常推荐使用–name来配置容器名，这样方便后续操作

启动容器

通过docker create启动的容器处于created状态，内部应用程序还没有执行，所以可以通过docker start命令来启动它。

当然我们可以通过docker run将上述的create和start两个操作合并成同一步

并且要注意的是，通常我们启动一个容器喜欢将其放到后台运行，而不是衔接到当前的控制台上，所以需要加一个参数-d，这个会告诉docker程序启动后在后台运行与控制台分离

docker run --name nginx -d ningx

管理容器

我们可以通过docker ps查看当前正在运行的容器的信息。如果要查看更加详细的信息加上-a选项

停止和删除容器

将正在运行的程序停止，通过执行docker stop即可。但是该操作，内部所做的修改仍然存在可以通过执行docker start继续执行

如果需要完全删除容器，可以通过docker rm命令将容器删除

如果是在运行中的容器默认是不可删除的，可以添加-f选项

一个比较好的实践是如果容器短期内不会使用，就要将其删掉

进入容器

通过docker exec进入容器，相当于我们可以通过docker exec帮助我们在容器里面执行命令

比较好的操作是通过docker exec命令来启动sh或者bash

-i表示保持输入流，使用它才能保证控制台程序能够正确识别我们的命令

-t表示启动一个为终端，形成与bash的交互，如果没有的话，没法看到bash内部的执行结果

衔接到容器

通过执行docker attach将当前的输入输出流连接到指定的容器

容器网络

沙盒Sandbox

提供给容器虚拟的网路栈，隔离了容器网络与宿主主机网络，形成了完全独立的容器网络环境

网路Network

Docker内部的虚拟子网，网络内的参与者相互可见并能够通讯

端点Endpoint

位于容器与网络隔离墙上的洞，其主要目的是形成一个可以控制的突破封闭的网路环境出入口

容器互联

在docker create或者docker run的时候创建--link选项进行配置

在webapp中只需要通过mysql这个名字就可以访问到

jdbc:mysql://mysql:3306/webapp,不需要知道具体的Ip

暴露端口

容器虽然运行了，但是并不代表可以随便访问。我们只能访问暴露出来的接口。

可以通过--expose选项暴露接口

端口映射

如果从容器外要访问容器提供的服务。我们可以把容器的端口映射到宿主操作系统的端口上，从外部访问宿主操作系统的端口，数据请求就会自动发送给与之关联的容器端口。

-p <ip>:<host-port>:<container-port>其中ip是宿主机器监听的ip，默认是0.0.0.0，host-port与container-port分别是宿主操作系统端口与容器的端口。这两者可以不一样。

通过docker ps，可以通过->标记了端口的映射关系

挂载方式

Docker提供三种试用不同场景的文件系统挂载方式

  • Bind Mount能够直接将宿主操作系统中目录和文件挂载容器内的文件系统中。通过指定容器外的路径和容器内的路径，就可以形成挂载的关系。在容器内外对文件的读写都是互相可见的。
  • Volume从宿主操作系统中挂载目录到容器内，只需要指定容器内的目录，不需要关系具体挂在到了宿主操作系统中的哪里
  • Tmpfs Mount支持挂载系统内存中一部分到容器的文件系统里去

挂载文件到容器

-v <host-path>:<container-path>hostpath，containerpath分别表示宿主操作系统中的目录和容器中的目录。

定义目录的时候必须使用绝对路径

通过执行docker inspect中的Mounts可以看到数据挂载相关的部分。

使用数据卷

数据卷的本质是宿主操作系统上的一个目录，这个目录存放在Docker内部，接受Docker管理

-v <name>:<container-path>

两个容器可以通过挂载相同的数据卷来共享数据

可以通过docker volume ls查看已经创建的数据卷

删除数据卷

可以通过执行docker volume rm volumename删除掉没有被引用的数据卷。

如果在创建数据卷的时候，没有命名，那么很难被删除掉。可以通过下面三种方法

  1. docker rm -v containername 通过删除容器一起把关联的数据卷删除掉
  2. docker volume rm 查看，找到没有关联的数据卷
  3. docker volume prune可以删除掉没有被容器引用的数据卷

数据卷容器

数据卷容器指的就是没有一个具体的应用，甚至不需要运行。主要是为了定义一个或者多个数据卷并持有它们的引用

--volumes-from打通了容器间文件系统的管理

数据卷备份与恢复

要备份数据卷，需要先通过一个临时容器，将要备份的目录和数据卷都挂载在这个容器上

恢复数据卷

保存和共享镜像

提交容器更改

将容器中修改的内容保存成镜像的命令是docker commit，提交之后可以在本地镜像里中找到docker images。可以在提交容器更改的时候给一个提交信息。

给镜像命名

提交更新后的镜像没有tag和repository

通过docker tag给镜像取名字

也可以通过docker tag对已经存在的镜像命名，但是两者引用这相同的镜像层。对应的image id一样。

也可以通过使用docker commit -m”…” container repository:tag

镜像迁移

通过执行docker save将镜像导出去

导入镜像

与docker save相对的是docker load

通过docker save和docker load同时能够导出和恢复多个镜像

导出和导入容器

可以直接通过docker export和docker import

Dockerfile

dockerfile包含了构建镜像过程中需要执行的命令和其他操作

dockerfile内容比较简单一种是注释行，一种是指令行

dockerfile的结构

  • 基础指令：用于定义镜像的基础和性质
  • 控制指令：描述镜像在构建过程中需要执行的命令
  • 引入指令：用于将外部文件直接引入构建镜像的内部
  • 执行指令：指定在启动的时候需要执行的脚本或者命令
  • 配置指令：对网络、用户等内容进行配置

常见的dockerfile指令

FROM

可以通过from指令指定一个基础镜像，接下来的指令都是基于这个基本镜像所展开

####RUN

RUN用于向控制台发送命令的指令，在RUN指令之后，我们可以拼接上需要的命令，在构建Docker就会执行这些命令，把对系统文件的修改记录下俩，形成镜像的变化

ENTRYPOINT和CMD

通过ENTRYPOINT和CMD实现在容器启动的时候，根据镜像所定义的命令启动容器中进程号为1的进程

EXPOSE

EXPOSE指定容器明确暴露的端口

VOLUME

直接帮助用户定义数据卷

构建镜像

通过执行docker build构建镜像

docker build后面接收的参数是本地路径或URL路径，不是dockerfile文件的路劲

Docker Compose

通过配置文件，将与应用系统相关的软件和他们对应的容器进行配置。

启动和停止

通过执行docker-compose up -d会根据配置内容，创建所有的容器、网络、数据卷等内容，并启动

通过执行docker-compose down来停止
