工作区&版本库

工作区就是电脑中可以看到的目录 例如我现在电脑下的learngit文件夹

版本库工作区下面有个.git文件 是Git的版本库

some keys:

git status

提示 untracked表示 从来没有被添加过

提示 modified表示修改了

其中版本库中有个很重要的概念是stage的暂存区

分解 添加文件到git仓库的步骤

  1. git add 把要提交的所有修改放到暂存区
  2. git commit 可以一次性把暂存区的所有修改提交到分支
版本管理

Git跟踪的是修改，而非文件

举个例子 ：如果你第一次修改了->git add->第二次修改->git commit

结果是只提交了第一次修改

撤销修改

场景1：修改了工作区里某个文件的内容，想要丢弃工作区里的修改时

这里有两种情况 一种是 在工作区中修改了，还没有放到暂存区里，现在撤销回到版本库中一模一样的状态 另外一种就是先前修改放到暂存区了，但是又进行修改，现在撤销回到暂存区一模一样的状态。

场景2：当你修改了工作区的文件，又添加到暂存区里，这个时候想要丢弃修改

场景3：当你不仅修改了，而且commit了

tips:

git checkout 其实使用版本库里的版本替换掉工作区的版本，可以做到“一键还原”

删除文件

场景：在工作区下删除文件 或者rm

可能1:确实是要删除的文件，这个时候版本库里并没有被删除，所以需要提交一下

可能2:误删了，现在需要从版本库中恢复

可能3：你通过git rm删除文件，这个时候添加到了暂存区，需要退回版本 然后在checkout

远程仓库

强大的github来了

首先需要建立ssh(在用户的目录下可以看到.ssh)然后在github上add ssh就可以将其关联起来，

在github下new repo 然后可以看到将这个repo clone到本地或者将本地的仓库push到repo

分支管理

分支的作用：你需要开发一个版本，但是这个版本的时间需要两周，如果你没写完就提交到master会导致别人无法进行操作，所以你需要自己开一个新的分支，然后把代码提交到这个分支上，当全部完成后在提交到master

创建和合并分支

由前面我们可以知道，git有个主分支master，master指向提交，HEAD指向当前分支。所以创建分支的操作为

  1. 创建一个新的分支,指向master相同的提交(可以得到当前的版本库中最新的版本–现在要分道扬镳了,我开发我的)
  2. 把HEAD指向这个新分支上，就表示当前分支

合并分支：

前面我们知道，创建一个新的分支，我们可以得到当前分支，然后后面的修改都在这个新的分支上，要合并分支，只要把master指向这个新的分支就可以了,并且删除这个新的分支(这里有个疑惑，不知道后面会不会解决掉。如果多人协作合并分支？)

tips:

如果在新分支上修改并且没有add commit，在master分支上也可以看到修改的内容，因为此时的修改是在工作区，仓库并不知道你对的是谁的修改。

但是一旦你add commit后就看不到了

解决冲突

当冲突无法自动合并的时候首先需要解决冲突，解决冲突之后，再提交，合并完成.

需要手动编辑解决冲突(上面那个问题解决了)，没有别的解决方案就是解决冲突

分支管理策略

Git合并分支会用Fast forward模式，这个模式下，删除分支，会丢失分支信息

Bug分支

场景：你正在dev分支上开发，但是突然接到需要修复一个bug，但是你此时还没有开发完，并且无法提交。

  1. 首先在dev分支上 git stash 将当前工作现场存储起来
  2. 在需要的分支下建立修复bug的分支
  3. 修复bug，合并分支
  4. 回到dev git stash pop恢复的同时把stash内容也删了
Feature分支

场景：你需要开发一个新功能，但是还没开发完需要对你刚才操作进行强制删除

  1. 开发新功能最好是新建一个分支feature
  2. 强制删除 git branch -D
多人协作

tips:

远程仓库的默认名为 origin

查看远程库的信息 git remote 查看更加详细的信息 git remote -v

推送分支 git push origin master 推送其他分支 git push origin dev

some key:

  1. master dev 分支需要推送
  2. bug分支其实可以不用推送
  3. feature分支看是否与人协作

场景：另外一个开发人员A需要跟你开发dev分支

  1. A需要在本地git clone仓库，不过默认情况下只能查看master分支(说好的一起开发？？)
  2. A在本地创建dev分支并且与远程绑定起来 git checkout -b dev origin/dev
  3. A开发推送(push) 你也开发推送，不过A快你一步，这个时候你push不上去因为发现有冲突
  4. 你需要git pull，但是先前没有指定dev与远程的origin/dev分支的连接，要设置两者之间的连接 git branch –set-upstream-to=origin/dev dev
  5. 你解决了冲突，然后提交上去
